# try18res_py12 改良点リスト

日経225トレーディング DQN（ResNet特徴抽出器）のコードレビュー。
重要度順に整理。`main.py` / `data.py` / `eval.py` / `run_simulation.py` などを対象。

## A. 重大なバグ・データリーク（最優先）

### A-1. 学習データと評価データの期間が重複している（リーク）
- 学習: `1997-01-01` 〜 `2024-01-01`（`main.py:375-378`）
- 評価: `2023-01-01` 〜 `2025-09-12`（`eval.py:69-70`）
- → 2023年が **学習期間に含まれたまま評価**されており、in-sample 評価になっている。
  パフォーマンス指標（年利・シャープ）は楽観的に過大評価される。
- **対策**: 学習・検証・テストを時系列で完全分離（例: 学習 〜2021、検証 2022、テスト 2023〜）。ウォークフォワード検証を推奨。
これは確か、７５日線のパラメータが必要用だったりwindowsizeが130ということがあり、重なってました

### A-2. `US_10Y_Rate` が実際は FF金利（米国10年債ではない）
- `data.py:17-18` で `FEDFUNDS`（フェデラルファンド金利）を取得し `US_10Y_Rate` に rename している。
- 特徴量名と中身が食い違っており、学習・解釈ともに誤解を招く。
- **対策**: 10年債を使うなら `DGS10` を使用。

### A-3. `trade_penalty` が定義だけで一切使われていない
- `main.py:96` で保持し `trade_count` も加算するが、**報酬から減算する処理が無い**（dead parameter）。
- 過剰トレード抑制の意図が機能していない。
- **対策**: `step()` 内で `if action != self.prev_action: reward -= self.trade_penalty` を追加するか、不要なら削除。
削除で良い

### A-4. 学習と評価で取引コスト・リスク制限が不一致
- `transaction_cost`: 学習 `0.0001`（`main.py:389`）vs 評価 `0.001`（`eval.py:81`）= **10倍差**。
- `risk_limit`: 学習 `0.6` vs 評価 `0.5`。
- → 学習時にコストを過小評価しているため、本番（高コスト）で過剰トレードになりやすい。
- **対策**: 現実的なコストに統一（往復スプレッド・手数料を含めた値）。設定を一箇所で共有。
コスト統一して

### A-5. 評価ループのステップ範囲が学習総ステップ数と不整合
- `eval.py:89` は `range(200000, 440001, 10000)` のモデルをロードするが、学習は `total_timesteps=350000`（`main.py:424`）。
- 350000 を超えるチェックポイントは存在せず、`load_model_safely` のフォールバックで**未学習モデル**を評価しかねない（例外を握り潰すため気付きにくい）。
- **対策**: 範囲を実在チェックポイントに合わせる。存在しないファイルはスキップしてログに明示。
35万にこれは合わせて

## B. 環境・報酬設計の問題

### B-1. リターン定義が混在している
- 残高更新は単純リターン `ret = (P_tomorrow - P_today)/P_today`（`main.py:145`）。
- 中期報酬は対数リターン `log(P_future/P_today)`（`main.py:174-177`）。
- 単純/対数が混在し、報酬スケールが不揃い。
- **対策**: どちらかに統一（対数リターン推奨）。
合わせて

### B-2. 中期報酬シェイピングが未来3日を毎ステップ加算
- `main.py:164-183` で `discount_factor=0.3` × 3日先リターンを **毎ステップ**報酬に上乗せ。
- 報酬が将来情報に強く依存し、TD学習の整合性を崩す（同じ将来区間を重複カウント）。
- **対策**: シェイピングを使うなら potential-based reward shaping にするか、`gamma` による割引に委ね、明示的な未来参照は外す。
修正して

### B-3. MinMax正規化が絶対水準情報を破壊・毎ステップ再計算
- `_get_observation`（`main.py:112-132`）は各ウィンドウごとに min-max 正規化。
  - 価格水準の情報が失われ、ウィンドウ間で基準がずれる（分布シフト）。
  - 毎ステップ `O(window×features)` で再計算しており非効率（window=130で重い）。
- **対策**: リターン化（差分/対数差分）＋ロバストな z-score、もしくは事前計算でキャッシュ。
対策して


### B-4. `daily_inflation = 0.0 / 365` のデッドコード
- `main.py:152-154` のフラット時インフレペナルティが常に 0。意図不明。
- **対策**: 機能させるか削除。
削除でいい

### B-5. エピソード終了時に観測を `None` で返す
- `main.py:202` `obs = ... if not done else None`。終端で `None` を返すのは VecEnv の自動リセットと相性が悪く、最終観測が欠落する。
- **対策**: 終端でも有効な観測（最後の obs）を返す。
やって

### B-6. 旧 `gym` API を使用（gymnasium 非対応）
- `import gym` / `reset()` が obs のみ・`step()` が 4-tuple（`main.py:102-206`）。SB3 2.7 は gymnasium（5-tuple, `terminated`/`truncated`）が標準で、shimmy 経由の暗黙変換に依存している。
- **対策**: `gymnasium` へ移行し、`reset()->(obs,info)`、`step()->(obs,reward,terminated,truncated,info)` に統一。
やって

### B-7. アクション定義のコメント不一致
- docstring/コメントで「0:Long 1:Flat 2:Short」と「0→Long,1→Flat」（`main.py:76`）が混在。
- **対策**: 定数（Enum）化して一元管理。
やって

## F. 運用ワークフロー上の困りごと（ユーザー報告）

### F-1. ローカルにGPUが無く、毎回 Colab を立ち上げる必要がある
- 学習だけ Colab(GPU)、評価・推論はローカル(CPU) という二重環境。
- **対策**:
  - `notebooks/train_colab.ipynb` を用意し、「リポジトリ clone → `uv`/`pip` で依存導入 → Google Drive を `/content/drive` にマウントして `models/` を Drive 保存」までを定型化。
  - チェックポイントを Drive に保存すれば、ローカルへ `rclone`/手動DLで取り込み、評価はローカルCPUで継続できる。
  - 依存は `pyproject.toml` に一本化（E-6）し、Colab 側は `pip install -e .` か `uv pip install -r` の1行で済ませる。
  - 学習設定を引数化（C-4）しておけば、Colab セルから window_size 等を渡して再現可能。

### F-2. リプレイバッファの警告が毎回出る
- 原因: `DQN` のデフォルト `buffer_size=1_000_000`。観測が `window=130 × features=28 × float32 ≈ 14.5KB/件` なので、満杯で **約14.5GB** 必要 → SB3 がメモリ不足警告を出す。
  （`eval.py:14-16` で警告を握り潰しているのは対症療法）。
- **対策**:
  - `buffer_size` を現実的な値へ（例 `100_000`〜`200_000`）。
  - `optimize_memory_usage=True`（next_obs を別管理せずメモリ半減）も検討。
  - 観測をリターン化・縮小（B-3）すれば 1件あたりも軽くなる。
  - 警告の filter ではなく根本のサイズ調整で消す。

### F-3. 10年債利回りが月次（1ヶ月おき）でしか変わらない
- 原因: 日本=`IRLTLT01JPM156N`（月次OECD）、米国=`FEDFUNDS`（月次, しかも10年債ではない／A-2）。日次 ffill で階段状になる。
- **対策**:
  - 米国10年債は **yfinance `^TNX`（CBOE 10年債利回り・日次）** に変更 → A-2 と F-3 を同時解決。FRED を使うなら `DGS10`（日次）。
  - 日本10年債の日次は無料ソースが乏しい。候補:
    - FRED `IRLTLT01JPM156N` のまま月次→**線形補間**で滑らかに（stale を緩和）。
    - もしくは日次系列が取れるソース（投資データAPI等）に差し替え。
  - 「金利は低頻度更新」と割り切る場合は、特徴量を「水準」でなく「変化があった事実/前回比」に変換するのも一案。

### F-4. 当日のデータをすぐ取得して投入できない
- 現状は `eval.py` / `run_simulation.py` で `manual_data` を手打ち（High=Low=Open=Close の合成行＝D-2 の歪みも誘発）。
- **対策**:
  - `data.py` に `fetch_latest()` を追加し、yfinance `Ticker("^N225").fast_info` / `history(period="5d")` で最新OHLC、`^VIX`・`^TNX` を自動取得。
  - 金利など低頻度値のみ引数 or キャッシュで補完。
  - Gradio(`server.py`) に「最新データ自動取得」ボタンを追加し、手打ちは上書き用途に限定。
  - 合成行ではなく実OHLCで延長 → ATR/TR の歪み（D-2）も解消。

### F-5. GPU使用量が 0.2GB 程度で学習が非効率
- 原因: DQNはオフポリシーで **1ステップ=1環境遷移**。ボトルネックは GPU ではなく **CPU側の env.step / 観測生成**（MinMaxを毎ステップ 130×28 再計算＝B-3）。GPU はほぼ遊んでいる。
- **対策（効果順）**:
  1. **観測の事前計算/キャッシュ（B-3）** … env ステップの最大の重しを除去。最も効く。
  2. **`batch_size` を 32→256/512、`gradient_steps`/`train_freq` を増やす** … 1回の収集に対し学習を厚くして GPU を使い切る。
  3. **複数環境で収集**（`SubprocVecEnv` で `n_envs=4〜8`）… データ収集スループット向上（DQNはオフポリシーなので多環境収集が有効）。
  4. ネット幅（`features_dim`）の拡大は最後の手段（CPUボトルネックが残ると無意味）。
  - 指標: `nvidia-smi`/TensorBoard(C-3) で `fps` と GPU util を見ながら調整。

### F-6. 複数モデルを一気に生成したい
- **対策**:
  - 学習を引数化（C-4）した上で `train_multi.py` を用意し、**シード/ハイパラのリストをループ**して `models/{exp_name}_{seed}/` に保存。
  - GPUが空いている（F-5）ので、1GPU上で **複数runを並列**起動可能:
    - 簡易: `subprocess`/`joblib.Parallel` で N プロセス同時起動。
    - スイープ管理が欲しければ Optuna / W&B Sweeps。
  - 出力命名を `name_prefix=f"{exp}_{seed}_{steps}"` に統一し、`eval.py` 側でグロブして横断比較。
  - アンサンブル運用（複数モデルの多数決）にもそのまま使える（`eval.py` の `final_actions` 集計と相性良）。

## G. 学習が不安定・利益が出ない問題（最重要・ユーザー報告）

> 症状: 学習が不安定、学習できたモデルでも平気でマイナス利益。安定して利益を出したい。
> DQN/ResNet 以外で安定した学習手段はないか。

### G-0. まず前提認識（重要）
- 金融データは **S/N比が極端に低い**（日次リターンはほぼランダムウォーク）。「不安定・負ける」のは設計ミスというより問題の本質も大きい。
- → **アルゴ/アーキの交換だけで黒字化はしない**。安定性は「①アルゴリズム ②報酬設計 ③正則化・検証 ④アンサンブル」の合わせ技で出る。アーキ変更の寄与は実はその中で一番小さい。
- 現状 DQN+BatchNorm(C-1)+壊れた報酬シェイピング(B-2)+in-sample評価(A-1) が重なっており、「不安定で負ける」のは半ば必然。まずここを直すのが先。

### G-1. DQN 以外のアルゴリズム候補（安定性順）
| アルゴ | 入手 | 行動空間 | 特徴 / 安定性 | 移行コスト |
|---|---|---|---|---|
| **QR-DQN**（分布型DQN） | `sb3-contrib` | 離散(現状のまま) | 報酬ノイズに強い分布型。**DQNからの差し替えが最小**で安定性UP。まず試すならコレ | ★低 |
| **PPO** | sb3本体 | 離散(現状のまま) | オンポリシーで**最も安定**・実装枯れ。サンプル効率は劣るが安定性最優先なら本命 | ★低 |
| **RecurrentPPO**（LSTM内蔵） | `sb3-contrib` | 離散 | 時系列の部分観測に強い。window特徴抽出を内蔵LSTMに任せられる | ★中 |
| **A2C** | sb3本体 | 離散 | PPOの軽量版。基本PPO優先 | ★低 |
| **SAC / TD3** | sb3本体 | **連続** | ポジションサイズを連続値(-1〜+1)で出せる→過剰トレード抑制。要 action空間変更 | ★高 |

- **推奨**: まず **QR-DQN**（最小移行で効果確認）→ 改善が頭打ちなら **PPO**。両方とも今の `NikkeiEnv` をほぼそのまま使える。
- 連続ポジションサイズに興味があれば SAC が筋が良い（「全力ロング/全力ショート」でなく強度を学べる＝ドローダウン低減）。

### G-2. ResNet 以外の特徴抽出アーキ候補
| アーキ | 特徴 | コメント |
|---|---|---|
| **TCN（Dilated Causal Conv）** | 因果的・長期依存に強く、RNNより安定して速い | 時系列では ResNet1D より素直。有力候補 |
| **LSTM/GRU** | 時系列の定番 | RecurrentPPO なら標準搭載。単体は学習やや不安定 |
| **Transformer** | 既に `TransformerFeatures` を実装済(未使用, C-5) | データ量少だと過学習しやすい。dropout必須 |
| **MLP（薄い）** | エンジニアリング済特徴に薄いMLP | **過学習が真の敵なら、簡素な方が汎化する**。ベースラインとして必ず比較すべき |
- **注意**: BatchNorm は廃止し LayerNorm/GroupNorm へ(C-1)。どのアーキでも dropout + weight decay を効かせる。
- **まず MLP ベースラインを取る**こと。複雑なResNet/Transformerがそれに勝てないなら、複雑さは過学習を生んでいるだけ。

### G-3. 「安定して利益」を出すための本質レバー（アーキより効く）
1. **報酬を risk-adjusted に変える** … 生の対数リターンでなく **差分シャープレシオ(Differential Sharpe Ratio, Moody&Saffell)** や「リターン − λ·ドローダウン/分散」。リスク調整報酬は学習を安定化し、勝ち方が一貫する。最重要。
2. **検証の徹底（A-1 と直結）** … ウォークフォワードで OOS 検証し、**検証スコアでモデル選択**。「学習できた=良い」ではない。
3. **アンサンブル（F-6 と直結）** … 複数シード/期間のモデルの多数決。単体の分散を平均化し、負けにくくなる。**最も手軽で効く安定化策**。
4. **正則化** … dropout・weight decay・小さめネット・早期終了。`exploration` と `gamma` の調整。
5. **ベンチマーク比較** … バイ&ホールド/常時ロングと必ず比較。それに勝てて初めて「edge あり」。今は勝てているか不明。
6. **複数シードで分布を見る** … 1回の結果で判断しない。5〜10シードの中央値と分散で評価。

### G-4. 推奨ロードマップ
1. **土台修正が先**: A-1(検証分離)・B-2(壊れた報酬)・C-1(BatchNorm)。これだけで不安定さは大きく減る。
2. **報酬を差分シャープに**(G-3-1) … 黒字の安定性に最も効く。
3. **アルゴを QR-DQN に差し替え**(G-1) … 同じenvで安定性比較。
4. **MLPベースライン vs ResNet/TCN を OOS で比較**(G-2)。
5. **5〜10シードのアンサンブル**(G-3-3, F-6) … 最終的な実運用はこれ。
6. 余力で PPO / RecurrentPPO / 連続SAC を試す。

> 依存追加: QR-DQN / RecurrentPPO は `sb3-contrib` が必要（`pyproject.toml` に追加）。
