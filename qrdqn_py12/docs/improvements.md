## B. 環境・報酬設計の問題

### B-3. MinMax正規化が絶対水準情報を破壊
- `_get_observation` は各ウィンドウごとに min-max 正規化。
  - 価格水準の情報が失われ、ウィンドウ間で基準がずれる（分布シフト）。
  - ※毎ステップ再計算の非効率は事前計算キャッシュで解消済み（B-3 / F-5）。残るのは正規化方式そのものの問題。
- **対策**: リターン化（差分/対数差分）＋ロバストな z-score を検討。

## F. 運用ワークフロー上の困りごと（ユーザー報告）

### F-4. 当日のデータをすぐ取得して投入できない
- 現状は `eval.py` / `run_simulation.py` で `manual_data` を手打ち（High=Low=Open=Close の合成行＝D-2 の歪みも誘発）。
- **対策**:
  - `data.py` に `fetch_latest()` を追加し、yfinance `Ticker("^N225").fast_info` / `history(period="5d")` で最新OHLC、`^VIX`・`^TNX` を自動取得。
  - 金利など低頻度値のみ引数 or キャッシュで補完。
  - Gradio(`server.py`) に「最新データ自動取得」ボタンを追加し、手打ちは上書き用途に限定。
  - 合成行ではなく実OHLCで延長 → ATR/TR の歪み（D-2）も解消。

## G. 学習が不安定・利益が出ない問題（最重要・ユーザー報告）

> 症状: 学習が不安定、学習できたモデルでも平気でマイナス利益。安定して利益を出したい。
> DQN/ResNet 以外で安定した学習手段はないか。
> ※ QR-DQN(G-1) と 差分シャープ報酬(G-3-1) は導入済み。以下は引き続きの方針・候補。

### G-0. まず前提認識（重要）
- 金融データは **S/N比が極端に低い**（日次リターンはほぼランダムウォーク）。「不安定・負ける」のは設計ミスというより問題の本質も大きい。
- → **アルゴ/アーキの交換だけで黒字化はしない**。安定性は「①アルゴリズム ②報酬設計 ③正則化・検証 ④アンサンブル」の合わせ技で出る。アーキ変更の寄与は実はその中で一番小さい。

### G-1. DQN 以外のアルゴリズム候補（安定性順）
| アルゴ | 入手 | 行動空間 | 特徴 / 安定性 | 移行コスト |
|---|---|---|---|---|
| **QR-DQN**（分布型DQN）※導入済 | `sb3-contrib` | 離散(現状のまま) | 報酬ノイズに強い分布型。**DQNからの差し替えが最小**で安定性UP | ★低 |
| **PPO** | sb3本体 | 離散(現状のまま) | オンポリシーで**最も安定**・実装枯れ。サンプル効率は劣るが安定性最優先なら本命 | ★低 |
| **RecurrentPPO**（LSTM内蔵） | `sb3-contrib` | 離散 | 時系列の部分観測に強い。window特徴抽出を内蔵LSTMに任せられる | ★中 |
| **A2C** | sb3本体 | 離散 | PPOの軽量版。基本PPO優先 | ★低 |
| **SAC / TD3** | sb3本体 | **連続** | ポジションサイズを連続値(-1〜+1)で出せる→過剰トレード抑制。要 action空間変更 | ★高 |

- **推奨**: QR-DQN で改善が頭打ちなら **PPO**。両方とも今の `NikkeiEnv` をほぼそのまま使える。
- 連続ポジションサイズに興味があれば SAC が筋が良い（「全力ロング/全力ショート」でなく強度を学べる＝ドローダウン低減）。

### G-2. ResNet 以外の特徴抽出アーキ候補 ※config切替を実装済
`config.FEATURES_EXTRACTOR` で切替（`resnet` / `tcn` / `lstm` / `gru` / `mlp`）。実体は `main.py` の各 `*Features` クラスと `make_features_extractor()`。

| アーキ | 特徴 | コメント |
|---|---|---|
| **TCN（Dilated Causal Conv）** | 因果的・長期依存に強く、RNNより安定して速い | 時系列では ResNet1D より素直。有力候補。`NUM_BLOCKS`/`TCN_KERNEL` で調整 |
| **LSTM/GRU** | 時系列の定番 | 単体は学習やや不安定。`RNN_LAYERS` で層数調整 |
| **MLP（薄い）** | エンジニアリング済特徴に薄いMLP | **過学習が真の敵なら、簡素な方が汎化する**。ベースラインとして必ず比較すべき。`MLP_HIDDEN` で幅調整 |
- **注意**: どのアーキでも dropout + weight decay を効かせる（今後の課題）。
- **まず MLP ベースラインを取る**こと。複雑なResNetがそれに勝てないなら、複雑さは過学習を生んでいるだけ。
- 残課題: OOS でのアーキ横断比較（G-3）。

### G-3. 複数OOS窓での評価（エッジ判定の前提・最重要）
- **現状の問題**: 評価が「2024-01〜2026-06」という **1つの強い上昇相場の窓だけ**。これは高分散の1回くじで、結論を出す土台になっていない。
  - 実際、ResNet/MLP ともシード・チェックポイント間の分散が大きく、単発の良い結果は運の引きでしかない。
  - val(2022-06〜2024-01, もみ合い〜軟調) と test(bull) の **レジーム不一致**で、検証選抜したモデルが test 下位に来る（検証スコアでの選抜が機能しない根本原因）。
  - この bull 窓では生リターンで B&H(+104%) に勝てたのは手選抜の外れ値モデルだけ。アンサンブルでも中央値≒B&H未満。
- **対策**: `_eval_one.py` を複数OOS窓に対応させ、**各窓ごとに B&H と比較**する。
  - 窓の例: 2020-22(コロナ) / 2022-24(軟調) / 2024-26(bull)。地合いの異なる窓を揃える。
  - 判定基準: 「**どの地合いでも B&H に対してプラスのエッジがあるか**」。1窓でなく窓×シャープ/超過リターンの分布で見る。
  - ここで初めて「エッジ有り／無し（=買い持ちが正解）」を判定できる。目的を「bullで勝つ」か「bear/暴落でDDを抑える」かで評価軸も決める。
- 詳細な実測ログは [[arch-comparison-mlp-vs-resnet]] を参照。
